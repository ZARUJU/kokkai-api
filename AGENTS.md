# AGENTS.md

このリポジトリで作業するエージェント向けの共通方針を定義する。
実装だけでなく、後から読んで理解しやすい状態を維持することを重視する。

## 基本方針

- 日本語で簡潔かつ丁寧に記述する
- データ取得元、整形内容、保存先が追えるようにする
- 一時的に動くだけでなく、後続の保守で読み返しやすい構成を優先する
- JSON 出力は人間が読める形式を保つ

## README 方針

- README はプロジェクトの入口として機能するように書く
- 少なくとも以下を含める
  - プロジェクトの目的
  - ディレクトリ構成
  - セットアップ方法
  - 各パイプラインの役割
  - 各パイプラインの入力、引数、出力先
  - 実行例
- 新しいパイプラインを追加したら README に追記する
- README の文章は説明不足でも冗長でもない状態を目指す

## パイプライン実装方針

- `src/pipeline` 配下の各ファイル先頭には、複数行のモジュールドックストリングを置く
- 先頭ドックストリングには少なくとも以下を書く
  - 何をするパイプラインか
  - 受け取る引数
  - 入力元
  - 出力先
  - 保存される主なデータ項目
- CLI 引数を取る場合は、`argparse` などで明示する
- 保存先パスは定数としてファイル上部に置く
- 実行時にどの JSON がどこにできるかをコード上で読み取りやすくする

## モデルとデータ構造

- JSON に保存する構造は `src/models.py` に Pydantic モデルとして定義する
- フィールド名は意味が明確な英語名を使う
- 元データの注記やカテゴリは、意味が落ちない範囲でフィールドとして保持する
- 日付は可能なら ISO 形式に正規化する

## 実装スタイル

- 共通処理は `src/utils.py` に寄せる
- HTML 解析は、表構造の揺れをある程度吸収できるように書く
- 取得元サイトの列構成差分がある場合は、条件分岐を隠さず明示する
- 公開関数と主要モデルには日本語 docstring を付ける

## 検証

- 変更後は少なくとも `uv run python -m compileall src` で構文確認する
- パイプラインを追加・変更した場合は、可能なら実データまたは保存済み HTML で一度動作確認する
- 実際に生成される JSON の先頭だけでも確認し、想定した構造になっているか見る

## このリポジトリで特に重視すること

- 「どこから取って、どう整形して、どこへ保存するか」が README とコード先頭だけで分かること
- 後から別の人や別のエージェントが見ても、追加修正の入口がすぐ分かること
